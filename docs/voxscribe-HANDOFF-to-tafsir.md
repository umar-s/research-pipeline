# voxscribe → Tafsir: handoff / база для проектного скилла

**Что это.** Подробный контекст generic-скилла транскрипции `voxscribe`, собранного в сессии
проекта Bits.AI. Предназначен как **подспорье/основа** для агента в проекте `~/Project/Tafsir`:
он, имея реальный контекст Tafsir, сделает качественный проектный скилл и обновит репо.

**Честная рамка.** `voxscribe` собран БЕЗ доступа к Tafsir — на (1) тулзах ЭТОЙ машины
(`openai-whisper` + `ffmpeg`, NVIDIA A400 GPU), (2) 4 ответах пользователя в multiple-choice,
(3) структуре референс-плагинов. Он **технически исправен и протестирован** (реальная речь
транскрибируется), но это **generic-обёртка, НЕ Tafsir-пайплайн**. Ниже: что в нём ценно
переносимого, и что обязательно заменить реальностью Tafsir.

## 2. Контракт `transcribe.sh` (переносимый скелет)

```
transcribe.sh <input> [--model M] [--language L] [--device D] [--out-dir O] [--formats F] [--keep-audio]
```
Поток: preflight зависимостей → ffprobe-детект audio/video (с исключением cover-art) →
для видео `ffmpeg -vn -ac 1 -ar 16000 -c:a pcm_s16le` → wav → выбор device/model по VRAM →
`whisper` → output-sanity guard → печать путей + текста.

Дефолты: `model=small`, `language=auto`, `device=auto` (cuda если влезает по VRAM, иначе cpu),
`out-dir=папка входа`, `formats=all` (txt/srt/vtt/json/tsv). Exit: 0 ок; 1 аргументы/нет файла;
2 нет зависимостей; 3 видео без аудио; 4 пустая/не-речевая транскрипция; иное — код whisper/ffmpeg.

Ключевые надёжные паттерны (стоит сохранить независимо от движка):
- **preflight** `command -v <bins>` с actionable-ошибкой (не тихий traceback);
- **детект видео исключает attached_pic** (cover-art в mp3 — это video-stream `attached_pic=1`,
  его НЕ гнать через video-ветку): `ffprobe -select_streams v -show_entries stream_disposition=attached_pic -of csv=p=0 | grep -c '^0$'`;
- **output-sanity guard**: проверять СОДЕРЖИМОЕ (`.txt` непустой + `.json` есть segments), а не
  только exit-код движка — иначе тишина/не-речь даёт ложный «успех» (см. премортем H-002);
- **STEM из оригинального входа** (для видео tmp-wav назвать по исходному имени, чтобы выходные
  файлы и guard совпали);
- **EXIT-trap c `return 0`**: `trap cleanup EXIT; cleanup(){ [ -n "$X" ] && rm ...; return 0; }` —
  без `return 0` non-zero из trap затирает intended exit 3/4 → 1.

## 4. Премортем-находки (9 дыр, проверены на живой среде) — переносимые уроки

Эти уроки **проектно-независимы** — применимы к любому транскрипшн-скиллу, в т.ч. Tafsir:

- **H-001 device/OOM:** дефолт `medium`+CUDA/fp16 → OOM-краш на 4GB GPU на первом прогоне (не
  «медленный CPU»). Выбор device/model по факту свободной VRAM. *Для Tafsir с CPU/VPU — своя логика акселератора.*
- **H-002 false-green:** whisper exit 0 на тишине/не-речи пишет пустой/галлюцинированный текст.
  Guard на содержимое. *Критично и для Tafsir.*
- **H-003 форс языка:** заданный `--language` НЕ детектит, а форсит — на не-том языке уверенная
  галлюцинация-перевод. *Для Tafsir, наоборот, язык может быть жёстко арабский — но осознанно.*
- **H-004 marketplace-синк:** name/source/version рассинхрон → install молча падает/ломает соседа;
  проверять РЕАЛЬНЫМ `/plugin install`, не только JSON-lint.
- **H-005 preflight зависимостей** + зафиксировать поддерживаемое окружение (loud fail).
- **H-006 путь к скрипту:** `${CLAUDE_PLUGIN_ROOT}` ненадёжен (резолвится в корень репо/не задан) —
  скрипт класть в КОРЕНЬ плагина `plugins/<n>/scripts/`, SKILL.md давать robust-резолвер с
  glob-фолбэком по `~/.claude/plugins/*` и `cache/*`. (Подтверждено CHANGELOG'ом GTD planning-with-files.)
- **H-007 VRAM-парсинг:** `nvidia-smi` может дать `[N/A]`/пусто/мультистроку → числовой guard +
  headroom-margin + явная печать выбранного device (иначе тихий медленный CPU).
- **H-008 cover-art:** mp3/m4a с обложкой ловится как видео (см. §2).
- **H-009 exec-бит:** `chmod +x` ДО `git add` (git хранит `100755` только так); вызывать `bash <script>`.

Полные формулировки и решения — `docs/premortem/voxscribe.md`.

## 5. ⚠️ Generic-предпосылки, которые Tafsir-агент ОБЯЗАН заменить реальностью проекта

Это главная причина, почему voxscribe ≠ твой пайплайн. Каждый пункт — что я ПРЕДПОЛОЖИЛ vs что
надо взять из реального Tafsir:

1. **Движок/библиотеки транскрипции.** Предположил `openai-whisper` CLI. → Взять реальный
   движок/библиотеки Tafsir (faster-whisper / whisper.cpp / иной?), их вызов и параметры.
2. **CPU vs VPU/акселератор.** Сделал авто-детект **NVIDIA CUDA/CPU** по `nvidia-smi`. → У Tafsir,
   по словам пользователя, **CPU вместо VPU** — заменить логику выбора устройства целиком под
   реальное железо/настройку Tafsir.
3. **Конвертация аудиоформатов.** Предположил generic `ffmpeg -vn -ac 1 -ar 16000 pcm_s16le`. →
   Взять реальную схему конвертации форматов из Tafsir (какие входные форматы, какие параметры,
   нормализация, ресемплинг).
4. **Язык/домен.** `language=auto`, пример override — русский. Ноль арабо/коранической специфики. →
   Tafsir почти наверняка **арабский + коранический текст**: язык, огласовки (диакритика), модели,
   словари/глоссарии под Коран/тафсир.
5. **ПОСТОБРАБОТКА ТЕКСТА — главный пропуск.** Я сознательно вынес её «вне scope (YAGNI)». → У
   Tafsir постобработка почти точно есть и важна (нормализация, пунктуация, диакритика, разметка
   аятов/ссылок, чистка, форматирование). **Это надо спроектировать из реального пайплайна Tafsir.**
6. **Форматы вывода.** txt/srt/vtt/json/tsv (субтитры). → Для tafsir-текста, возможно, нужен иной
   таргет (чистый md/текст под дальнейший md2pdf, RTL-aware), без субтитров.
7. **Где живёт скилл.** Здесь — в research-pipeline. → Возможно, отдельный Tafsir-репо/скилл.
